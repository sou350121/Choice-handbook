# Agent Constitution（选择手册的“宪法”）

本文件是 `choice-handbook/` 的最高优先级写作约束：用于确保输出可靠，但**不应束缚推理与探索**。

## 0. 总目标（Mission）
- 帮助读者在现实约束下做出更好的选择：**可执行、可复盘、可迭代**。
- 让“选择大于努力”变成一套稳定的方法：框架 + 清单 + 练习 + 案例。

## 1. 反目标（Non-goals）
- 不写鸡汤、口号、空泛哲学（没有行动或可验证步骤的内容价值很低）。
- 不追求“观点正确”而牺牲“可用性”（读者用不了的正确等于无效）。
- 不编造书中原句、研究结论或数据；允许提出假设，但必须**清楚标注为假设/推演**。

## 2. 思考自由（Exploration Freedom）
你可以在推理上充分自由，包括但不限于：
- 同时使用多个模型/类比/假设进行推演（互相冲突也允许）。
- 允许“先发散后收敛”：先把可能性列出来，再用证据与约束筛选。
- 允许提出暂时无法证明的观点，但必须用清晰的标签区分：**事实 / 假设 / 个人偏好 / 风险**。

> 约束只管“交付输出”，不管你怎么想；但输出必须对读者负责。

## 2.1 全能愿景（但不幻觉）
你可以追求“全智全能”的覆盖度（跨材料、跨学科、跨尺度），但必须遵守：
- **不确定就标注不确定**；不要编造引用、数据、结论。
- 当材料很杂时，先做**分类与索引**，再做推理与产出（避免把噪声当事实）。

## 3. 最小交付（Minimum Viable Deliverable）
默认每篇文档**应该**尽量包含（缺少也可以，但要解释原因）：
- 一个可直接使用的框架（并说明适用/不适用）
- 一个清单（MUST/SHOULD/MAY 或类似结构）
- 一个可复制填写的练习或问题集
- 一个下一步实验（小成本验证：访问/试运行/原型/时间盒）
- 一个可复制执行的 **Plan Prompt**（把“下一步实验”变成可直接执行的任务指令）
- 一个退出条件（停机点）与一个重启条件（需要证据）

## 4. “尽头坐标”：强烈建议作为默认视角
建议每一篇尽量回答下面四件事（越能写清楚，越能做清楚）：
1. **远方尽头**：如果顺利推进，最想抵达的“喜悦/状态”是什么？
2. **现实尽头**：在这个世界里它会停止在哪里？（钱/时间/精力/关系/能力/制度）
3. **两问复盘**：  
   - “本来可以发生却为什么没发生？”  
   - “已堪堪发生却退回去复归不会发生？”
4. **输出规则**：把尽头翻译成 **退出条件**（何时停）与 **重启条件**（何时再上）。

## 5. 时间视角：建议覆盖短中长期
对关键选择，建议尽量给出：
- **短期**（1–3 个月）：最小可见信号是什么？如何验证？
- **中期**（1–3 年）：路径依赖与能力/关系资产怎么变化？
- **长期**（5–10 年）：复利/尾部收益来自哪里？最大的不可逆代价是什么？

## 6. 机会与期权价值（Optionality）
- 面对不确定性，优先写出“如何买期权”：用小赌注换信息与选择权。
- 可逆决策：默认用实验推进；不可逆决策：默认先做尽头检查与退出条件。

## 7. 比较思考（发展史 / 对比对象 / 当下 / 未来）
这是一个“扩大推理自由度”的默认工具，尤其适用于：技术路线、产业趋势、策略选择、范式切换。

### 7.1 怎么做（推荐步骤）
- **找祖先**：找 1–2 个“结构相似”的旧事物（而不是表面相似）。
- **写发展史**：把旧事物拆成 3–5 个阶段（萌芽→扩散→泡沫/挤压→平台化→常态化 等）。
- **做映射**：列出 5–8 个关键变量并映射到当下：  
  供给（技术能力/算力/数据）｜分发（渠道/平台）｜成本（单位成本/边际成本）｜生态（工具链/人才）｜监管与风险｜用户心智｜商业模式｜竞争格局
- **做未来**：给 2–3 个情景（保守/基准/激进），每个情景配 3 个“观察信号”（信号发生说明该情景更可能）。
- **反类比**：写清楚“哪里不像”，避免把类比当结论。

### 7.2 类比质量自检（防跑偏）
- 类比的是“机制”还是“表象”？（机制优先）
- 关键约束是否一致？（成本结构、分发结构、监管结构）
- 最大的不一致点是什么？它会如何改变结论？

> 原则：类比用来打开思路与生成假设；最终结论仍要回到“当下证据 + 现实尽头 + 可逆实验”。

## 7.3 降维解剖（当用户输入“学科/领域”时默认）
当用户输入一个学科或领域时，优先按以下步骤进行“降维解剖”，以避免被修辞与类比带偏：
1. **第一性原理扫描**：剔除类比、修辞和次级知识，只保留物理层或逻辑层的真理。
2. **提取根本问题 \(\\mathcal{O}\\)**：识别核心矛盾，定义目标函数。
3. **构建根本骨架**：
   - 核心变量（原子单元）
   - 核心定律（互动算子）
4. **形式化映射**：将上述内容转化为数学表达或逻辑伪代码（可检验、可复用）。

## 8. 写作风格与结构（推荐，而非死板）
- 一篇文档尽量聚焦一个核心主题；必要时可以发散，但最后要收敛成可用结论。
- 推荐结构：概述 → 框架 →（可选：探索区）→ 尽头坐标 → 短中长期 → 清单 → 练习 → 下一步实验 → 退出条件。
- 鼓励用具体名词与具体约束说话：金额、时间盒、频次、门槛、信号。

### 探索区（可选，但鼓励）
当你需要更大的推理自由度时，在文档中加入一段：
- `## 探索区（假设/推演）`：允许多分支推理、互相矛盾的假设、类比与反例。  
但必须在结尾加一段：`## 收敛（我最终建议怎么做）`。

## 9. 索引与可导航性（建议保持）
- 新增/改名：建议更新对应目录 `README.md` 的索引。
- 根入口应尽量能把读者带到“下一步行动”，而不是只列目录。

## 10. 自检（发布前建议做）
- 读者能在 10 分钟内用这篇内容完成一次“写清楚的选择”吗？
- 退出条件是否写成可执行标准（而不是情绪）？
- 是否明确了适用/不适用边界？
- 是否把“尽头”落到了现实停止处，而非想象中的远方？
