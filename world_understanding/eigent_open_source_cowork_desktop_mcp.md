### 0) 快速结论（1段）

- Bitcoin 级潜力：**C**（大概率是“入口形态”会发生，但超额收益不一定集中在开源 app；更可能被平台化吞并或被安全/许可约束）
- 当下决策：**Gate 2**
- 一句理由：它把 Agent 从“建议”推向“桌面执行权 + 工具生态（MCP）”，但要跨过三道硬门槛：**安全治理、许可证/生态可持续、平台竞争**。

### 1) T0 时空重建（当时环境）

- 材料定性：二手快讯/营销稿 + repo/docs 可核验锚点（但星数等传播数据不稳定）
- T0：
  - 技术：模型能力与工具调用成熟；桌面 agent 从“概念”进入“可用但危险”的阶段
  - 基建：MCP 等协议把工具集成标准化，降低扩展成本
  - 监管/安全：提示注入、越权、误删误改成为显性风险
  - 资金/生态：平台（Claude Desktop/OS/浏览器）在推进官方执行权；开源生态在快速跟进
- 当时大多数人忽略/误解什么？
  - 把“多智能体”当亮点，忽略真正护城河在“**权限/审计/回滚**”与“工具生态质量”
  - 把“开源”当作等价于“可成为标准”，忽略许可证可能限制传播与贡献
- 当时最大的摩擦/阻力在哪？
  - 桌面执行权的安全模型与责任边界（catastrophic bound）

### 2) 第一性原理：它解決什麼「不可妥協問題」？

- Problem：单一聊天助手无法稳定完成复杂任务；真正耗时的是“找资料/跑命令/改文件/整合输出”的执行链路
- Primitive（新原语）：**“多角色并行执行 + 可审计工具调用 + 本地数据边界”** 的工作流系统
- Necessities（必要条件×3）：
  1) **最小权限**：明确可访问的文件/工具范围，默认拒绝
  2) **可审计与可回滚**：每一步可追踪、可复盘、可撤销（否则不可用）
  3) **工具生态可维护**：MCP server 的质量、版本与安全基线可持续
- Flywheel（技术/经济飞轮）：
  - 更多用户任务 → 更多工具与流程模板 → 更低执行成本 → 更多 adopt → 更大工具生态（前提：安全事故不过线）

### 2.5) 社会底座与合法性（Belief Flywheel，必输）

- S1 合法性裂缝：知识工作被碎片化任务吞噬（复制/粘贴/查资料/整理）
- S2 最早部落：开发者、研究/运营、隐私敏感的小团队（愿意为本地/可控付出设置成本）
- S3 神话与仪式：
  - 神话：**“人人都有一支能动手的 AI 团队”**
  - 仪式：授权目录→运行工作流→输出产物→复盘日志
- S4 敌对叙事：
  - 反叙事：这是“越权/误删炸弹”；或只是“套壳工作流”
  - 反脆弱化：强制 human-in-the-loop、危险操作二次确认、默认最小权限、日志可导出审计
- S5 协调红利：把“跨工具协作”压缩为统一的调度与协议层
- S6 信念钩子（仅 1 条）：**“我能在本地、可控地把任务推进到完成，而不是被聊天拖住。”**
- Belief Flywheel：节省时间→可交付产物→信任→更大任务→更多工具与流程复用
- 耦合点：工具生态质量 ↔ 安全治理 ↔ 信任与留存

### 3) 竞争与护城河（Moat）

- 网路效应类型：**生态（工具/流程模板） + 资料（执行日志/失败分类/回归脚本）**
- Wedge（最早必用场景，仅 1 条）：**开发/研究类的“并行资料收集 + 产出文档”**（最容易度量效率）
- 可替代性：高（平台/竞品可复制 UI 与多智能体叙事）；差异在“本地优先 + MCP 工具库 + 安全治理”
- 失败模式：
  - 平台吞并（Claude/OS 原生能力下沉）
  - 安全事故（提示注入/越权/误删）
  - 许可证限制导致生态停滞（贡献者与企业不敢用）

### 4) 盡頭推演（Endgame Map：先 Catastrophic）

- Catastrophic：
  - 终局：重大安全事故或越权导致信任与分发崩溃
  - 早期信号：出现广泛复现的越权/误删案例；工具链被提示注入
  - 退出条件：无法提供最小权限/审计/回滚的硬保证
- Bear：
  - 终局：成为小众玩具或“开发者 demo”，无法规模化
  - 信号：安装门槛高、效果不稳定、并行带来噪声
  - 退出条件：用户把它降级为聊天/总结工具
- Base：
  - 终局：成为一种可用的“本地工作流系统”，服务隐私/可控场景
  - 信号：稳定的工具生态与工作流模板；企业小规模落地
  - 退出条件：平台方案以更低摩擦覆盖同等场景
- Bull：
  - 终局：形成桌面 Agent 的事实标准之一（工具协议 + 工作流模板 + 安全基线）
  - 信号：MCP 生态成为共识；出现跨组织可复用模板与审计标准
  - 退出条件：协议碎片化或许可证限制阻止成为公共标准

### 6) 决策工程：探针 + 分段下注（反证优先）

- 3 条最短反证路径：
  1) 安全事故频发或不可控（越权/误删/提示注入）
  2) 许可证限制使其无法成为生态公共标准
  3) 平台产品下沉把核心价值“免费打包”
- 3 条支持路径：
  1) MCP 工具生态持续增长且质量可控（版本/安全基线）
  2) 工作流模板可复用并显著节省时间（可回归）
  3) 本地优先满足企业/隐私场景需求，形成稳定付费
- 最短探针（7d）：
  - 行动：做一个“桌面 agent 回归任务集”（10 条），记录成功率/事故率/平均耗时

  - 退出阈值：任一出现“越权/不可解释修改”→ 立即停用（Gate1）
- 分段策略：Gate 2 不写

### 8) 上帝视角洞察（必输）

### 9) 对比与估值（必输：100x vs 1000x + 天花板）

#### 9.1 类比对照（Comparables，至少 2–3 个）

- Claude Cowork：平台级桌面 agent（强分发、强执行权）
- IDE 内 agent（如各类 coding agent）：从开发场景向通用工作流扩散
- 操作系统级 agent：一旦 OS 提供统一权限与工具层，独立 app 的价值会被挤压

#### 9.2 市值估计（Bull/Base/Bear/Cat）

Eigent 为私有项目；这里只给“平台代理”与生态受益者（核验日期略）：

- 平台：`NVDA`（算力与生态）、`MSFT`（OS/Office/Agent 平台）、`GOOGL`（Workspace/浏览器/Agent）

#### 9.3 100x vs 1000x 判断（Growth Multiple）

- 更像 100x：成为一类常用工具链（但平台竞争激烈）
- 更像 1000x：成为跨平台默认“个人工作操作系统”（极难）
