### 2) 第一性原理：它解決什麼「不可妥協問題」？

- Problem：行业重复造轮子，百万研发费里大量消耗在复刻基础功能与工程坑上 → 速度慢、成本高、经验私有化
- Primitive（新原语）：**“可复现全栈参考设计（硬件+软件+调试SOP+测试矩阵）”** 让人形研发第一次更像软件工程（可复盘/可回归/可协作）
- Necessities（必要条件×3）：
  1) **复现资料完整**：BOM/替代件/装配/调试/安全阈值齐全且可执行
  2) **版本治理可用**：接口稳定、模块边界清晰、回归测试可跑
  3) **社区贡献闭环**：外部复现的失败分类与修复能回流主线（否则碎片化）
- Flywheel（技术/经济飞轮）：
  - 更多复现者 → 更多失败分类/修复 → 文档/测试矩阵更完整 → 复现成功率上升 → 更多团队愿意站在此底座上做差异化 → 更多贡献回流

### 2.5) 社会底座与合法性（Belief Flywheel，必输）

- S1 合法性裂缝：**“闭源黑箱 + 私有手艺”** 让行业整体速度被重复劳动拖慢（公共基础设施缺口）
- S2 最早部落：高校实验室/开源爱好者/早期机器人初创/系统集成商（他们愿意用时间换可复制底座）
- S3 神话与仪式：
  - 神话：**“人形机器人也该有 Linux/ROS 级的公共底座”**
  - 仪式：按版本复现、跑测试矩阵、提交失败分类与 Know-how PR
- S4 敌对叙事：
  - 反叙事：开源只是营销；复现会翻车；安全责任不可控；核心能力不可能开源
  - 反脆弱化：把“翻车”变成资产——公开失败分类、测试阈值与回归路径（越多失败越能加固底座）
- S5 协调红利：降低“从 0 到跑”的协作成本，让陌生团队共享同一基线，缩短迭代与对照时间
- S6 信念钩子（仅 1 条）：**“我能照着做复现出来，并在同一底座上二开，而不是永远从零造轮子。”**
- Belief Flywheel：
  - 疲劳/不信任（闭源魔术）→ 开源复现成功案例 → 社会证明（stars/复现报告）→ 更多贡献与合作 → 更高复现成功率
- 耦合点：repo 活跃度/复现报告/测试矩阵完善度 ↔ 复现成功率 ↔ 更多团队采用

### 3) 竞争与护城河（Moat）

- 网路效应类型：**生态/协议（接口标准） + 资料（失败分类/测试回归数据）**
- Wedge（最早必用场景，仅 1 条）：**教学/研究/早期产品团队的“可复现基线”**（先把腿跑起来，再谈上层能力）
- 可替代性：纯技术上可被复制；但“可复现链路 + 测试矩阵 + Know-how 知识库 + 治理”组合才难复制
- 失败模式：
  - 碎片化：fork 众多但主线不吸收，生态分裂
  - 安全/责任事故：导致声誉与监管 catastrophic
  - 供应链不可复现：BOM 替代困难，复现成功率低，信念飞轮断裂
  - 商业闭环不清：无法持续投入维护与认证

### 4) 盡頭推演（Endgame Map：先 Catastrophic）

- Catastrophic：
  - 终局：安全事故/专利许可纠纷/监管介入 → 项目被迫收缩或失去合法性
  - 早期信号：出现严重伤人/财产事故；关键专利与许可争议升级；维护者失联
  - 退出条件：主线长期停更 + 复现报告大面积失败 + 安全阈值缺失或不可执行
- Bear：
  - 终局：沦为“学习玩具/展示工程”，难进入产业交付链
  - 信号：外部复现多但二开无法沉淀；没有稳定测试矩阵与版本治理
  - 退出条件：外部贡献长期无法合并回主线；模块接口频繁破坏
- Base：
  - 终局：成为“人形机器人开发的通用参考基线”（类似 ROS 在机器人软件层的地位）
  - 信号：高校/开源社区持续复现；有稳定的发布节奏与回归测试；出现多家基于此二开的衍生项目
  - 退出条件：出现更强的替代底座，且迁移成本低、生态更强
- Bull：
  - 终局：底座 + 测试/认证 + 交付服务（JDM）形成行业事实标准
  - 信号：出现“认证版本/一致性测试”；产业伙伴用其缩短交付周期；形成“复现成功率”口碑
  - 退出条件：无法建立标准与认证，或被更强标准吞并

### 6) 决策工程：探针 + 分段下注（反证优先）

- 3 条最短反证路径：
  1) **复现不可用**：外部团队按文档无法跑通（失败率高且不可定位）
  2) **主线失活**：维护节奏断裂，issue/PR 无响应，版本治理缺失
  3) **碎片化**：fork 大量分叉，接口不兼容，知识库不回流
- 3 条支持路径：
  1) 外部复现报告增多且成功率上升
  2) 测试矩阵逐步标准化（耐久/抗摔/速度等）并形成回归
  3) 产业合作出现可复盘交付案例（JDM/工程化交付）
- 最短探针（7d）：
  - 行动：对 `Roboparty/roboto_origin` 做一次“可复现审计”（BOM/替代件/装配/调试/测试矩阵/版本发布/issue 响应）
  - 可观测信号：最近 30–90 天 commit/issue 速度；外部复现报告/PR；文档是否覆盖“失败分类与恢复”
  - 退出阈值：若 repo 长期停更（例如 >60 天无有效维护）且外部复现问题无闭环 → 降级为“材料收藏”，停止跟踪
- 分段策略：Gate 2 不写（仅买信息期权）

### 9) 对比与估值（必输：100x vs 1000x + 天花板）

#### 9.1 类比对照（Comparables，至少 2–3 个）

- ROS/ROS2：软件层公共底座 → 价值捕获多发生在集成/交付/硬件与服务，而非开源本体
- Linux：公共底座加速行业，但商业价值沉到发行版、服务、生态与标准化
- Open Compute / 参考设计：把硬件工程标准化，压低重复劳动，价值向供应链与规模化制造迁移

#### 9.2 市值估计（Bull/Base/Bear/Cat）

本项目是开源底座 + 初创团队叙事，缺少可核验财务数据；这里不做“单点估值”，只做可交易的**公开代理**：

- **ETF（核验 2026-01-21）**：`BOTZ`（Global X Robotics & AI ETF）、`ROBO`（ROBO Global Robotics & Automation ETF）
- **产业链代理（核验 2026-01-21）**：
  - 机器人本体/应用：`9880.HK`（UBTECH）、`1810.HK`（小米，作为“生态/资本背书代理”，非等价）
  - 上游自动化：`6954.T`（FANUC）、`6506.T`（Yaskawa）、`ABBN.SW`（ABB）
  - 关键部件（减速器）：`6324.T`（Harmonic Drive Systems）、`6268.T`（Nabtesco）
  - 工业机器人曝光：`TER`（Teradyne，含 Universal Robots）

> 注意：这些是“主题代理”，不是对 RoboParty 的直接投资标的。

#### 9.3 100x vs 1000x 判断（Growth Multiple）

- 更像 100x 的条件：它成为事实标准，并在“认证/交付/制造共同定义（JDM）”形成可扩张利润池
- 更像 1000x 的条件：需要跨行业标准化 + 强合法性/安全治理 + 全球生态（难度极高）
- 若证据不足：维持 Gate 2，只做信号板与最短探针

### 10) 发展路线图像（必须像图）

```
Phase0  开源发布
  - 里程碑：仓库/文档/基础BOM上线
  - 信号：基础可跑 + 最小装配SOP
  - 失效：资料不全/无法复现
```
