## 12.x 平台战争（06 硬化案例）

### 12.1 iOS（成功） vs Symbian（失败）
- **Outcome**：iOS 以统一体验+开发者生态胜出；Symbian 在碎片化与生态中出局。
- **T0 环境**：智能机从硬件竞争转向“应用分发+SDK+支付”。
- **Killer Assumption**（Symbian）：运营商/硬件渠道会永远决定胜负。
- **Wedge**：App Store + 统一 SDK（开发者 ROI 清晰）。
- **Flywheel**：更好 SDK/分发 → 更多开发者 → 更多应用 → 更多用户 → 更强分发。
- **Moat**：账号/支付/应用库沉淀（迁移成本）。
- **First Tripwire**：开发者开始“先 iOS 后其他”；关键应用只在 iOS 首发。
- **Catastrophic Bound**：生态供给断裂（关键应用缺失）且无法在 1–2 个版本周期内补齐。
- **Death Spiral**：应用少 → 用户少 → 开发者更少 → 应用更少。
- **Noise**：单款机型硬件参数。
- **最短探针（7d）**：看“供给侧”：开发者工具、分发摩擦、关键应用覆盖率与更新节律。

### 12.2 Android（成功） vs Windows Phone（失败）
- **Outcome**：Android 以 OEM 生态扩张；WP 生态不足+策略迟疑。
- **T0 环境**：多厂商生态竞争，默认预装与渠道权重极高。
- **Killer Assumption**（WP）：靠品牌与 UI 能在缺生态的情况下逆转。
- **Wedge**：免费授权 + OEM 合作 + Google 服务绑定。
- **Flywheel**：更多 OEM → 更多设备 → 更多用户 → 更多开发者 → 更强 OEM 意愿。
- **Moat**：预装与服务绑定、开发者生态。
- **First Tripwire**：关键应用缺失成为常态；开发者 ROI 不成立。
- **Catastrophic Bound**：开发者与 OEM 同时撤离（双边市场同时塌）。
- **Death Spiral**：应用缺失 → 用户不买 → OEM 不推 → 应用更缺失。
- **Noise**：少数评测口碑。
- **最短探针（48h）**：列“关键应用清单”与“关键 OEM 合作清单”，缺任何一条都先 Gate 2。

### 12.3 Chrome（成功） vs Internet Explorer（失速）
- **Outcome**：Chrome 以快速迭代+标准+开发者工具赢；IE 被安全与标准债务拖垮。
- **T0 环境**：Web 成为应用平台，开发者体验决定生态。
- **Killer Assumption**（IE）：预装份额足以长期锁定。
- **Wedge**：自动更新 + 速度/安全 + DevTools。
- **Flywheel**：更好工具 → 开发者更愿意适配 → 网站体验更好 → 用户增长 → 更强标准影响力。
- **Moat**：标准制定影响力 + 与搜索/服务协同。
- **First Tripwire**：开发者开始把 IE 当“兼容负担”。
- **Catastrophic Bound**：安全事故+标准不兼容导致企业/用户迁移潮。
- **Death Spiral**：开发者不适配 → 体验差 → 用户迁移 → 开发者更不适配。
- **Noise**：短期份额。
- **最短探针（7d）**：看“开发者抱怨的方向”与“标准实现速度”。

### 12.4 Microsoft（Win 时代成功） vs Netscape（失败）
- **Outcome**：OS 分发决定早期浏览器战争；Netscape 被切断分发。
- **T0 环境**：默认入口在 OS；用户不主动安装。
- **Killer Assumption**（Netscape）：技术领先足以抵消分发劣势。
- **Wedge**：捆绑+默认。
- **Flywheel**：默认入口 → 份额 → 开发者适配 → 更强默认正当性。
- **Moat**：OS 控制权。
- **First Tripwire**：OEM/渠道开始被迫预装竞争者。
- **Catastrophic Bound**：分发入口被封死（无法预装、无法默认）。
- **Death Spiral**：份额↓ → 开发者不适配 → 体验↓ → 份额更↓。
- **Noise**：产品单点功能。
- **最短探针（48h）**：画“分发地图”：默认入口在哪里？谁控制？能否被切断？

### 12.5 Facebook（成功） vs MySpace（失败）
- **Outcome**：Facebook 以更强产品/工程与关系链治理赢；MySpace 因体验债务与治理失速。
- **T0 环境**：社交网络是留存驱动型产品，质量劣化会迅速反身。
- **Killer Assumption**（MySpace）：流量与广告收入可以替代体验。
- **Wedge**：实名关系链 + feed 推荐。
- **Flywheel**：更好内容分发 → 更高留存 → 更多内容供给 → 更强分发。
- **Moat**：关系链与内容图谱（迁移成本）。
- **First Tripwire**：垃圾内容/体验崩坏导致留存下降。
- **Catastrophic Bound**：关系链迁移（用户在新平台重建社交）。
- **Death Spiral**：体验差 → 用户走 → 内容更差 → 广告更侵入 → 更差体验。
- **Noise**：短期 PV。
- **最短探针（7d）**：看留存与内容质量指标（举报率/屏蔽率/负反馈）。

### 12.6 Steam（成功） vs “缺社区与资产沉淀的商店”（失败）
- **Outcome**：Steam 靠分发+社区+账号资产沉淀形成护城河。
- **T0 环境**：数字分发降低边际成本，差异来自社区与资产。
- **Killer Assumption**（输家）：只要拿到独占/补贴就能赢。
- **Wedge**：库资产 + 成就/好友/创意工坊。
- **Flywheel**：更多用户 → 更多开发者 → 更多内容与社区 → 更多用户。
- **Moat**：账号资产与社交（迁移成本）。
- **First Tripwire**：开发者不愿把更新/社区放在你的平台。
- **Catastrophic Bound**：内容供给断裂且用户资产不可沉淀。
- **Death Spiral**：内容少 → 用户少 → 开发者少 → 内容更少。
- **Noise**：短期促销 GMV。
- **最短探针（30d）**：看“开发者投入深度”：是否愿意把 DLC/UGC/社区运营放进来。

### 12.7 AWS（成功） vs “价格战云”（失败）
- **Outcome**：AWS 用规模+生态+可靠性胜；价格战云在资本周期逆转时崩。
- **T0 环境**：云是资本密集型基础设施，成本曲线决定生死。
- **Killer Assumption**（输家）：补贴能永久换来留存。
- **Wedge**：可组合 API + 开发者入口。
- **Flywheel**：规模↑→单位成本↓→更低价/更好产品→更多客户→规模更↑。
- **Moat**：成本曲线+生态+迁移成本。
- **First Tripwire**：单位成本不下降、可靠性事故频发。
- **Catastrophic Bound**：融资窗口关闭仍需要高资本开支续命。
- **Death Spiral**：可靠性差/成本高 → 客户走 → 规模下降 → 成本更高。
- **Noise**：短期大单。
- **最短探针（7d）**：算单位经济学：毛利、资本开支强度、可靠性指标与迁移成本。

### 12.8 支付网络（成功） vs “只做钱包”（失败）
- **Outcome**：支付赢在场景与网络效应；纯钱包缺场景难闭环。
- **T0 环境**：双边市场（商户/用户）需要冷启动策略。
- **Killer Assumption**（输家）：只要用户多就会自然有商户。
- **Wedge**：线下商户网络+补贴节奏。
- **Flywheel**：更多商户 → 更多可用场景 → 更多用户 → 更多商户。
- **Moat**：风控/合规+商户网络。
- **First Tripwire**：欺诈率上升、监管收紧、商户拓展停滞。
- **Catastrophic Bound**：合规/牌照被卡导致网络断裂。
- **Death Spiral**：风控差 → 欺诈↑ → 监管↑ → 商户/用户流失 → 风控更差。
- **Noise**：下载量。
- **最短探针（30d）**：看商户留存、交易频率与风控成本。

### 12.9 开放标准（成功） vs 封闭标准（失败）
- **Outcome**：开放标准更易成为默认层；封闭标准在生态竞争中被兼容层/替代品击穿。
- **T0 环境**：生态价值>单点控制。
- **Killer Assumption**（封闭方）：控制力能长期换来生态。
- **Wedge**：兼容性与迁移路径。
- **Flywheel**：更多实现 → 更多用户/开发者 → 更强默认。
- **Moat**：标准地位与互操作。
- **First Tripwire**：出现兼容层绕过、开发者开始“默认不用你”。
- **Catastrophic Bound**：关键伙伴离开，标准失去多实现。
- **Death Spiral**：实现少 → 采用少 → 更少实现。
- **Noise**：短期授权收入。
- **最短探针（48h）**：数“多实现”与“互操作性”是否在增加；否则先 Gate 2。

### 12.10 Slack（成功） vs “单点协作工具”（被平台捆绑吞并）
- **Outcome**：协作工具若不能占据 SoR，会被平台（套件）以捆绑与默认入口挤压；Slack 通过生态与集成存活并被高价收购。  
- **T0 环境**：企业协作是强分发与强集成市场（入口在邮件/日历/身份/文件）。  
- **Killer Assumption**（输家）：只要功能更好，企业就会迁移（忽略套件捆绑与迁移成本）。  
- **Wedge**：团队沟通“高频入口” + 大量集成（把自己插进工作流）。  
- **Flywheel**：更多集成/机器人 → 更多团队使用 → 更强工作流依赖 → 更多集成。  
- **Moat**：工作流嵌入深度（集成/自动化）+ 组织惯性。  
- **First Tripwire**：平台把同类功能变默认并“免费”；IT 采购开始把你当冗余。  
- **Catastrophic Bound**：平台同时控制身份/文件/会议入口，且你无法成为 SoR（只剩聊天壳）。  
- **Death Spiral**：捆绑替代 → 新增放缓 → 开发者生态减少 → 集成减少 → 更难留存。  
- **Noise**：短期活跃与“口碑爆棚”。  
- **最短探针（7d）**：列出你在客户里的“不可替代集成/自动化”清单；若只是聊天/通知，风险很高。

### 12.11 Google Search（成功） vs Yahoo（相对失败：入口错位）
- **Outcome**：Google 以搜索质量+广告闭环成为互联网入口；门户把入口押在内容聚合，最终失去默认入口地位。  
- **T0 环境**：信息爆炸，用户需要“意图驱动”的导航，而不是编辑驱动的频道。  
- **Killer Assumption**（门户）：用户会永远从首页开始。  
- **Wedge**：更高质量搜索（更快找到答案）。  
- **Flywheel**：更好搜索 → 更多查询 → 更好数据与广告匹配 → 更多收入 → 更好基础设施。  
- **Moat**：分发（默认搜索）+ 数据闭环 + 广告网络。  
- **First Tripwire**：用户开始绕过门户首页，直接用搜索框；搜索成为浏览器默认入口。  
- **Catastrophic Bound**：默认入口（浏览器/OS）被搜索占领后，门户失去“起点”。  
- **Death Spiral**：入口流失 → 广告变弱 → 内容投入下降 → 更难留住入口。  
- **Noise**：频道内容短期流量。  
- **最短探针（30d）**：画“入口路径”：用户从哪里开始？默认在哪里？你是否能控制默认？

### 12.12 Apple App Store（平台税/规则） vs Epic（冲突：被切分发风险）
- **Outcome**：当平台控制分发与支付时，它能通过规则与抽成重塑生态利益；与平台正面冲突会承受“分发被切”的极限风险。  
- **T0 环境**：移动端分发集中，应用无法绕过商店触达用户；支付与合规（审核）是平台权力来源。  
- **Killer Assumption**（被平台约束者）：你可以在平台上做大到足以“反过来逼平台让步”，但平台仍掌握默认入口。  
- **Wedge**：分发入口（上架/推荐/搜索）+ 支付通道（抽成）。  
- **Flywheel**：平台：更多用户→更多开发者→更多应用→更强控制权→更易调整规则；开发者：规模越大→越依赖平台→越难退出。  
- **Moat**：平台的默认入口与审核权；开发者唯一的护城河是“可迁移的用户关系与跨平台分发”。  
- **First Tripwire**：平台开始用规则/抽成对你的单位经济学产生决定性影响；或出现“选择性执法/审核不确定性”。  
- **Catastrophic Bound**：被下架/支付被切断（分发归零），且没有可用替代渠道。  
- **Death Spiral**：规则收紧/抽成↑ → 单位经济学恶化 → 试图绕开 → 冲突升级 → 分发受限 → 收入下滑 → 更依赖平台。  
- **Noise**：短期舆论胜负、诉讼阶段性结果。  
- **最短探针（7d）**：写“平台依赖度”清单：分发占比、支付占比、用户关系是否可迁移（邮箱/账号/跨平台登录）；若不可迁移，默认高风险。

### 12.13 TikTok（成功） vs 旧社交（相对失速：分发机制被改写）
- **Outcome**：当分发从“关系链”迁移到“算法内容流”，新的分发机制能快速吞噬注意力；旧机制会在同质化下失速。  
- **T0 环境**：内容供给爆炸，用户更愿意消费“无关系链的高匹配内容”。  
- **Killer Assumption**（旧社交）：关系链是不可动摇的护城河，算法只能做增量。  
- **Wedge**：更强的内容匹配与创作者冷启动分发。  
- **Flywheel**：更高消费时长→更多反馈数据→更好推荐→更高时长。  
- **Moat**：推荐系统与创作者生态（供给侧）+ 分发效率。  
- **First Tripwire**：用户时长增量开始集中到新分发形态；创作者 ROI 向新平台迁移。  
- **Catastrophic Bound**：供给侧迁移导致内容质量差距不可逆（你再也追不上供给与数据）。  
- **Death Spiral**：时长下降→供给下降→内容质量下降→时长更下降。  
- **Noise**：短期活动拉新与补贴。  
- **最短探针（30d）**：用 3 个指标看“分发机制迁移”：时长、创作者产出/收入、内容冷启动效率（新号能否起量）。

### 12.14 Microsoft Teams（平台捆绑） vs Zoom（点产品：被套件挤压风险）
- **Outcome**：当协作入口被“套件化”并绑定身份/文件/会议，点产品即使体验更好也会被采购与默认入口挤压。  
- **T0 环境**：企业软件的默认入口在身份（SSO）与套件（邮件/日历/文件），采购更偏向整套而非单点。  
- **Killer Assumption**（点产品）：只要体验更好就能长期赢（忽略默认入口与捆绑定价）。  
- **Wedge**：套件的默认入口（预装/默认会议链接/组织通讯录）+ “顺手就有”的迁移路径。  
- **Flywheel**：套件：更多席位→更多默认使用→更强组织惯性→更多席位；点产品：新增变慢→生态投入下降→更难新增。  
- **Moat**：套件的身份/文件/日历 SoR 位置与捆绑定价。  
- **First Tripwire**：客户采购把你当“重复功能”；平台开始把同类能力变默认并免费。  
- **Catastrophic Bound**：平台控制关键入口（身份/会议/文件）且你无法成为 SoR（只剩功能差异）。  
- **Death Spiral**：捆绑替代 → 新增放缓 → 生态/集成减少 → 迁移成本下降 → 更易被替代。  
- **Noise**：短期口碑与评测领先。  
- **最短探针（7d）**：写“入口依赖度”表：新增渠道里有多少来自套件默认入口？若平台改默认，你的新增会掉多少？

### 12.15 Elastic（收紧许可） vs OpenSearch（Fork：生态分裂）
- **Outcome**：当开源项目收紧许可以防云厂商“白嫖”，会触发 fork 与生态分裂；结果可能是双生态长期竞争。  
- **T0 环境**：云平台具备分发与托管优势；开源商业化与生态信任高度耦合。  
- **Killer Assumption**（收紧方）：收紧许可不会伤害社区与默认地位。  
- **Wedge**：分发入口（云托管）+ 兼容性（API/插件）。  
- **Flywheel**：fork 方：托管/分发→更多用户→更多贡献→更强默认；原方：商业化→投入→产品更强→留住默认。  
- **Moat**：生态默认（插件/运维习惯/兼容性）与品牌信任。  
- **First Tripwire**：社区贡献下降、合作伙伴观望、云平台推出兼容 fork 并给迁移工具。  
- **Catastrophic Bound**：默认生态（插件/集成/托管渠道）迁移到 fork，原项目失去“默认”。  
- **Death Spiral**：收紧许可 → 社区分裂 → 迁移工具与兼容层出现 → 默认迁移 → 更分裂。  
- **Noise**：短期争论与舆论站队。  
- **最短探针（30d）**：看三件事：\n  - 兼容层/迁移工具是否成熟\n  - 主要托管渠道把谁当默认\n  - 插件/集成生态增量流向哪里

### 12.16 Terraform（收紧许可） vs OpenTofu（Fork：标准争夺）
- **Outcome**：基础设施工具一旦成为事实标准，许可变更会引发 fork；真正的战争在“生态默认与兼容性”。  
- **T0 环境**：IaC 工具嵌入交付与审计关键路径（SoR 属性强），迁移成本高但一旦触发会很彻底。  
- **Killer Assumption**（收紧方）：用户不会迁移，因为迁移成本太高。  
- **Wedge**：治理与可信承诺（未来规则是否稳定）+ 兼容性（HCL/Provider）。  
- **Flywheel**：fork：信任承诺→贡献上升→兼容持续→更多采用→更强默认。  
- **Moat**：Provider 生态与组织流程嵌入深度。  
- **First Tripwire**：大客户/云平台/发行版开始把 fork 当默认；Provider 维护者表态。  
- **Catastrophic Bound**：关键 Provider 与发行版只支持 fork（原项目生态被抽空）。  
- **Death Spiral**：许可不确定 → 信任下降 → 贡献转移 → 生态增量转移 → 默认被重写。  
- **Noise**：短期口号与“反正都能用”。  
- **最短探针（7d）**：对任何“基础设施 SoR”项目，强制问：未来 3 年规则会不会变？如果会，谁承诺兼容？承诺是否可信？

### 12.17 Redis（收紧许可） vs Valkey（Fork：默认托管与生态归属）
- **Outcome**：当核心基础组件收紧许可，云厂商与社区可能联合 fork 以维持“开放默认”；胜负取决于托管渠道与生态默认。  
- **T0 环境**：基础组件高度平台化（托管/发行版）；用户更在意“可预期的规则”而非单点性能。  
- **Killer Assumption**（收紧方）：默认地位不会因许可变化而动摇。  
- **Wedge**：托管渠道默认（云/发行版）+ 兼容性与迁移工具。  
- **Flywheel**：默认托管→更多采用→更多贡献→更强默认。  
- **Moat**：生态默认与稳定治理。  
- **First Tripwire**：主要发行版/云托管把 fork 当默认；生态贡献转移。  
- **Catastrophic Bound**：fork 成为“事实默认”，原项目变成小众分支。  
- **Death Spiral**：许可变化 → 发行版/托管切换 → 用户迁移 → 生态增量迁移 → 默认更固化。  
- **Noise**：短期争论与版本号。  
- **最短探针（30d）**：观察“默认渠道”：云托管、Linux 发行版、主要框架/SDK 的默认选择。

### 12.18 Twitter API（规则变更：第三方生态被关门）
- **Outcome**：当平台突然提高 API 成本/限制权限，依赖生态会被“一夜清零”，第三方产品直接归零。  
- **T0 环境**：第三方依赖平台数据与权限，且缺乏可迁移的用户关系与替代数据源。  
- **Killer Assumption**：平台不会对核心 API 改变规则（或会提前很久沟通）。  
- **Wedge**：权限与配额（rate limit/price）是平台的生杀开关。  
- **Flywheel**：平台：通过限制把价值内化；第三方：无法留存→无法继续投入→更依赖平台。  
- **Moat**：平台的身份与数据 SoR；第三方只有“自有用户关系/自有数据”才有护城河。  
- **First Tripwire**：API 定价/权限口径开始频繁变化；平台推出原生替代功能。  
- **Catastrophic Bound**：API 权限被收回或成本上升到单位经济学不可行。  
- **Death Spiral**：规则收紧 → 功能受限/成本上升 → 用户流失 → 收入下降 → 更无力应对 → 更流失。  
- **Noise**：短期“谈判/例外”传闻。  
- **最短探针（48h）**：对任何依赖平台 API 的产品，强制写：若 API 价格翻 10 倍或权限砍 50%，你还能活吗？替代数据源是什么？

### 12.19 Chrome Manifest V3（平台规则：扩展生态被重塑）
- **Outcome**：平台通过技术与政策改变扩展能力边界，直接重塑生态可行性与商业模式（例如广告拦截能力）。  
- **T0 环境**：浏览器是平台，扩展依赖其 API；平台既是裁判也是选手（广告/隐私）。  
- **Killer Assumption**：平台会永远允许扩展做“平台不喜欢但用户喜欢”的事。  
- **Wedge**：API 能力边界（能否拦截/修改请求、性能预算）。  
- **Flywheel**：规则改变→扩展能力下降→用户体验变化→扩展生态重排→平台更易推进自身策略。  
- **Moat**：平台默认入口与审核权；扩展只能靠多平台分发与独立渠道。  
- **First Tripwire**：平台开始以“安全/性能/隐私”之名收紧能力；审核不确定性上升。  
- **Catastrophic Bound**：关键能力被移除，扩展失去核心价值主张（等于被判死刑）。  
- **Death Spiral**：能力收紧 → 价值下降 → 用户下降 → 生态投入下降 → 更难维持价值。  
- **Noise**：短期政策讨论与时间表反复。  
- **最短探针（7d）**：把价值主张映射到 API：如果关键 API 消失，你还有什么可交付？有无替代平台/渠道？  

### 12.20 iOS vs Flash（平台禁用关键运行时）
- **Outcome**：平台禁用插件/运行时，生态被迫迁移；Flash 退出移动平台。  
- **T0 环境**：平台控制运行时与分发渠道，开发者依赖跨平台运行时降低成本。  
- **Killer Assumption**：平台会长期允许第三方运行时占据体验关键路径。  
- **Wedge**：安全/性能/电量等理由成为“规则改写”的正当性。  
- **Flywheel**：禁用运行时 → 开发者转向平台原生/标准 → 生态更依赖平台 → 平台更敢改规则。  
- **Moat**：平台对运行时与分发的双重控制。  
- **First Tripwire**：SDK/审核规则开始限制插件能力；平台公开倡导替代标准。  
- **Catastrophic Bound**：平台彻底禁用运行时或无法上架。  
- **Death Spiral**：被禁 → 开发者迁移 → 用户流失 → 生态更弱 → 被禁更彻底。  
- **Noise**：短期“兼容层/过渡期”承诺。  
- **最短探针（7d）**：列出核心功能依赖的运行时/API；若被禁用，能否在 1 个版本周期内迁移？  

### 12.21 Android vs Fire OS（开源可用但系统服务缺失）
- **Outcome**：没有核心服务与默认分发的 fork 生态难以成形，应用供给不足。  
- **T0 环境**：Android 开源，但关键体验依赖 GMS/Play 服务与分发默认。  
- **Killer Assumption**：有代码就有生态，开发者会自动跟随。  
- **Wedge**：系统服务（地图/推送/支付）与分发默认不在你手里。  
- **Flywheel**：关键应用缺失 → 用户少 → 开发者 ROI 低 → 应用更少。  
- **Moat**：系统服务 + 默认分发渠道。  
- **First Tripwire**：Top 应用不上架、更新滞后、兼容性问题频发。  
- **Catastrophic Bound**：关键应用生态断裂（用户无法完成核心任务）。  
- **Death Spiral**：应用少 → 用户少 → 生态投入少 → 应用更少。  
- **Noise**：短期“安装包/侧载方案”。  
- **最短探针（30d）**：做“关键应用覆盖率+更新时效”清单，低于阈值即判定生态失败。  

### 12.22 iOS ATT（平台隐私规则重写广告生态）
- **Outcome**：平台强制隐私同意，广告归因与投放效率下降，生态结构重排。  
- **T0 环境**：移动广告高度依赖 IDFA 与跨应用追踪。  
- **Killer Assumption**：平台不会切断关键数据与归因能力。  
- **Wedge**：隐私规则变更（强制 opt-in）使可追踪用户大幅减少。  
- **Flywheel**：归因下降 → 广告 ROI 下降 → 预算收缩 → 流量变贵/变少 → 归因更差。  
- **Moat**：平台掌握系统级权限与默认设置。  
- **First Tripwire**：opt-in 率持续低于阈值、CPM 上升、ROI 恶化。  
- **Catastrophic Bound**：核心广告模型单位经济学失效。  
- **Death Spiral**：ROI 下降 → 预算下降 → 流量变少 → ROI 更差。  
- **Noise**：短期“替代归因方案”宣传。  
- **最短探针（48h）**：量化 IDFA 依赖度：若可追踪用户下降 70%，你的 CAC/LTV 还能活吗？

## 模式总结（06 版）：平台战争的通用“死法”和最快探针

### A) 反复出现的 Killer Assumption（致命假设 Top 7）
- **功能胜于入口**：忽略默认入口（预装/默认/套件）对增长与采购的决定性。  
- **平台规则稳定**：忽略平台会通过抽成、审核、API、能力边界“改写你的单位经济学”。  
- **用户关系可迁移**：实际上用户关系/身份/支付在平台手里，你没有可迁移资产。  
- **生态会自然长**：缺少开发者 ROI 与工具链，生态不会自发繁荣。  
- **兼容层不会出现**：低估 fork/兼容层/迁移工具会把你的护城河“拆解成可复制部件”。  
- **捆绑不会发生**：低估平台用“免费+默认”把你定义为重复功能。  
- **平台不会亲自下场**：忽略平台原生功能吞并（native feature absorption）的概率与速度。  

### B) 经典 Death Spiral（最常用的箭头链条）
- **入口螺旋（Distribution Spiral）**：默认入口被占 → 新增下降 → 生态投入下降 → 价值下降 → 更难新增  
- **捆绑螺旋（Bundling Spiral）**：平台免费/默认 → 采购把你当冗余 → 新增变慢 → 生态变弱 → 更冗余  
- **规则螺旋（Rule Spiral）**：规则收紧/抽成↑ → 单位经济学恶化 → 试图绕开 → 冲突升级 → 分发受限 → 更恶化  
- **生态抽空螺旋（Ecosystem Drain）**：贡献/伙伴转移 → 迁移工具成熟 → 默认迁移 → 增量更向新默认集中  

### C) 最快探针（48h/7d/30d）
- **48h：分发地图**：默认入口在哪里？谁控制？有没有“一键切断”的生杀开关？  
- **48h：平台依赖度表**：分发占比、支付占比、身份/数据/关系是否可迁移。  
- **7d：生态供给侧**：关键应用/关键集成/关键开发者是否把你当“默认优先”？  
- **7d：规则风险清单**：抽成/审核/API/能力边界的最坏情景与替代路径。  
- **30d：fork/兼容层风险**：迁移工具成熟度、托管渠道默认、发行版/云/套件的默认选择。  

### D) 用这份库写 Gate（建议）
- **默认 Gate 2**：只要你还没写出“分发地图 + 平台依赖度表 + 规则最坏情景”。  
- **不得 Gate 3**：若任一条成立：分发可被切断且无替代渠道 / 用户关系不可迁移 / 平台已原生覆盖或即将覆盖你的核心价值。
