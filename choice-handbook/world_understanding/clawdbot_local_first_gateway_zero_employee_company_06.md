## 1) T0 时空重建（当时环境）

- 材料定性（先贴标签，避免拿错工具）：
  - 类型：repo（可核验）+ 媒体传播稿（强叙事、弱证据）
  - 阶段：爆红扩散期（stars 快速增长；但数字强时变）
  - 可信度：架构/安全默认以 repo/README 为准；社媒“钱包清 0”等仅作风险提示
  - 时效：强（每天变）

- **T0-Now（此刻当时）**：
  - 技术：Node ≥22；本地 Gateway 常驻（daemon）；工具执行/多通道接入；可选 sandbox
  - 基建：个人设备（macOS/Linux/Windows via WSL2）+ 消息通道（WhatsApp/Telegram/iMessage 等）
  - 监管：执行器接触真实账号/消息面；安全事故（若发生）更像“合规/信任”硬约束
  - 资金：开源 MIT；分发与采用更像“口碑/开发者网络”驱动
  - 社会心理：从“聊天”转向“动手”；“0 员工公司/AI 员工”成为叙事符号

- **T0-Analog（类比回放，必输）**：
  - 选 3 个结构类比物：
    - A) **Zapier/IFTTT（自动化）**：从“手动操作”到“工作流自动化”
    - B) **Hubot/ChatOps（聊天驱动运维）**：把命令与执行权搬到聊天里
    - C) **TeamViewer/远程桌面（远程操控）**：把“执行权”远程化后的安全与信任门槛
  - 早期领先指标（真的信号）：
    - A：能稳定跑完高频工作流、失败可复盘（幂等/队列/告警）
    - B：能把“执行权”放进团队协作而不失控（权限/审计/回滚）
    - C：远程暴露后仍能把事故率压到可接受阈值（默认最小权限）
  - 早期噪声/宣传（易误导）：

    - “0 员工公司”更多是叙事压缩，不是商业闭环证明
  - 常见死法（失败模式）：
    - 被安全事故击穿信任（凭证外泄/误操作）
    - 被平台/OS 原生能力下沉吞并（分发与迁移成本不足）
    - 可靠性不足导致“能 demo 但不可依赖”

    - 下一阶段最小里程碑：**把“远程执行权”变成默认可控：pairing/allowlist/sandbox/审计导出/回滚/危险操作二次确认** 能在真实场景长期稳定

## 2) 第一性原理：它解決什麼「不可妥協問題」？

- Problem：人不可能 7×24h 坐在电脑前，但任务与沟通随时发生；“远程执行权”缺少低摩擦入口。
- Primitive（新原语）：**本地常驻 Gateway（控制平面）+ 多通道收发 + 工具执行流 + skills 扩展**。
- Necessities（必要条件×3）：
  1) 默认最小权限（unknown senders 不可执行；危险工具可关/可隔离）
  2) 审计与回滚（能复盘：谁让它做了什么、做错了怎么撤回）
  3) 可靠性工程（daemon/重连/队列/幂等；否则“睡一觉帮你做完”不可成立）
- Flywheel（技术/经济飞轮）：
  - 更多可复用任务 → 更多 skills/workflows → 更低摩擦的自动化 → 更高留存 → 更多贡献/生态 → 更强分发

## 2.5) 社会底座与合法性（Belief Flywheel，必输）

- S1 合法性裂缝：人对“信息过载 + 执行碎片化 + 隐私与账号被平台控制”的不满上升
- S2 最早部落：builders/极客/效率党/独立开发者（愿意忍受 setup 摩擦）
- S3 神话与仪式：神话一句话——“你拥有一个随叫随到的本地执行器”；仪式——每天把待办丢给它跑
- S4 敌对叙事：提示注入/钱包清零/越权执行（“执行器=高危后门”）
- S5 协调红利：把“指挥-执行”从同一屏幕解耦到多通道；降低等待与切换成本
- S6 信念钩子（仅 1 条）：**“它在我的机器上、我能控制它的边界。”**
- Belief Flywheel：不想被拴在屏幕前 → 渴望分身 → 传播 demo → 社会证明/星标 → 更多人安装 → 更强信念
- 耦合点：执行成功率/事故率 → 口碑传播；一次严重事故也会反向放大

## 3) 竞争与护城河（Moat）

- 网路效应类型：**生态/资料/协议**（skills 与通道/工具适配形成生态惯性）
- Wedge（最早必用场景，仅 1 条）：**“离开电脑也能让它在本地替你跑完一组明确任务”（下载/整理/总结/提交/通知）**
- 可替代性：高（许多 agent 框架都能做）；差异在 **默认安全、通道覆盖、onboarding 体验、可靠性与生态**。
- 失败模式：最可能死于
  - 安全/信任崩（提示注入/凭证外泄/远程暴露配置错误）
  - 分发被平台下沉（OS/浏览器/IM 平台原生能力整合）
  - 可靠性不足（任务队列/幂等/可回归做不好）

## 4) 盡頭推演（Endgame Map：先 Catastrophic）

#### 最可能的尽头（必输：短/中/长）

- **短期（0–12 个月）**：
  - 最可能发生：爆红扩散 + 大量“实测” + 安全实践被迫产品化（pairing/sandbox/doctor）
  - 最早停机点：一次可复现的大规模安全事故（或媒体化事件）把“执行器”叙事打回 Gate 1
  - 6 个月社会需求雷达：
    - 一线需求：异步执行、账号/消息统一入口、本地隐私边界、可控远程访问
    - 二线需求：skills 市场、回归任务集、审计导出/事故归因、沙箱默认化
    - 深层需求：个人工作“控制平面”（像个人 OS 层）
    - T0-Analog 映射：
      - 2020（远程协作爆发）：需求从“沟通”转向“可交付与可审计”
      - 2010–2012（移动平台起跑）：入口迁移先发生，killer app 往往滞后
      - 2023（LLM 普及）：从“会说”转向“会做”，安全成为硬约束
      - 3 条可验证预测（含最早信号）：
        1) 一年内“默认 sandbox/allowlist”会从可选变成默认建议（信号：文档/doctor 强制提示更重）
        2) skills 将出现“签名/来源信任/权限声明”标准化（信号：官方文档新增强制门禁）
        3) 平台原生能力会下沉同类形态（信号：OS/IM 平台推出更低摩擦的执行入口）
- **中期（1–3 年）**：
  - 稳态：两极化——要么成为“个人工作控制平面”的一类产品形态，要么退回极客玩具
  - 卡点：安全与可靠性工程（把事故率压到可接受阈值）
- **长期（3–10 年）**（四象限）：
  - Catastrophic：被定义为高危后门形态（重大事故/监管/平台封锁）
  - Bear：被平台能力商品化，开源项目边缘化
  - Base：小众但稳定的本地助手（重视隐私的用户群）
  - Bull：成为个人/团队的默认执行入口（控制平面标准化 + 生态繁荣）

#### 停止机制（必输）

1) 出现可复现的“默认配置也能被提示注入导出凭证”的链路（必须降级/暂停）
2) 远程暴露（Tailscale/Funnel/SSH 等）导致的入侵事件高频出现
3) 可靠性长期不可回归：任务成功率低、误操作率高且无法通过回归体系持续改善

#### 「本来可以发生却没发生」（必输）

- “人人都有 AI 分身、全自动跑工作流”可能没发生的核心阻力：**信任与灾难上界**（执行器一旦越权/误操作，代价比“聊天幻觉”大得多）。

#### 「如果成功」的终极状态（必输）

- 每个人都有一个本地常驻的“工作控制平面”，能在多个通道接收指令并执行。  
- 权限/审计/回滚像今天的密码管理器一样变成默认配置。  
- 组织里大量“执行岗位”被自动化，但治理（审计/回归/权限）反而成为核心岗位。

## Hard Forcing Functions（不满足就不得 Gate 3）

1) Killer Assumption（致命假设）：默认配置下能把“执行器的灾难上界”压到可接受阈值
2) First Tripwire（最早坏信号）：一次可复现的越权导出（私钥/cookie/凭证）
3) Catastrophic Bound（不可跨红线）：执行器导致不可逆资产/账号损失或大规模数据外泄
4) Wedge（最早必用场景）：离开电脑也能让本地机器完成一组明确任务并回报结果
5) Flywheel（自增机制）：更多任务→更多 skills/workflows→更强留存与贡献→更强分发
6) Death Spiral（死亡螺旋）：事故→媒体化→信任崩→平台封锁/用户流失→维护萎缩→更多事故
7) Belief Hook（信念钩子）：“它在我的机器上，我能控制它的边界。”
