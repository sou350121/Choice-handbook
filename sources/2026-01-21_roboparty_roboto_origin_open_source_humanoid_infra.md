## 2) 可抽取资产（Definitions / Variables / Mechanisms）

### 2.1 定义（这篇材料给出的“可复用概念”）

- **基础设施型开源**：不止开源算法或旧版本，而是开源“硬件图纸 + 电控原理 + 采购清单 + 组装/调试 SOP + 训练/控制栈 + 测试矩阵 + 知识库”。
- **可复现链路**：让“第一次上手”的不确定性下降到“照着做就能成”的工程路径（或至少把失败变成可定位）。
- **Know-how 共创知识库**：把隐性调试经验（避坑、顺序、力矩参数等）显性化，并允许社区补充。

### 2.2 关键变量（后续验证/反证会用到）

- **复现成本**：BOM 与替代件可得性、供应链波动、最低可复现批量。
- **复现成功率**：不同团队/不同渠道零件/不同装配能力下的成功率与失败类型分布。
- **迭代速度**：新动作/新策略/新硬件替换的“从改动到稳定跑起来”的周期。
- **社区活跃度**：commit 频率、issue/PR 响应时间、外部贡献比例、fork 的“有效二开”数量。
- **测试矩阵质量**：测试指标是否可执行、是否覆盖耐久/抗摔/高速等、是否给出明确阈值。

### 2.3 机制（为什么“开源全栈”可能改变行业节奏）

- **重复劳动 → 公共底座**：基线越标准化，边际创新越集中到差异化能力而非脚手架。
- **复现成本下降 → 人才与团队密度上升**：更多人能进入同一基线，形成更快的对照/复盘与迭代。
- **知识库与测试矩阵 → 工程科学化**：把“经验”转成可回归、可检验、可复制的流程资产。

## 3) 关键主张（Claims）与可验证预测（Predictions）

> 这些不是事实，是材料在暗示的“可被验证/反证”的东西。

1) **“全栈可复现”比“局部开源”更能加速行业**  
   - 预测：同一套底座被高校/实验室/初创反复复现，且复现周期随版本迭代显著下降。
2) **模块化（ROS2 包架构）能显著降低二开摩擦**  
   - 预测：外部贡献主要发生在独立模块（硬件/部署/训练），并能通过标准接口合并回主线。
3) **sim2real gap 的系统性方案会成为“复现成功率”的关键瓶颈项**  
   - 预测：能给出清晰的 sim2real 迁移步骤、参数与失败分类的项目，会形成更强的生态吸引力。
4) **商业化不靠“绑定零件”，而可能靠“交付/制造共同定义（JDM）与服务”**（材料提及）  
   - 预测：后续会出现可核验的企业客户案例：从参考样机 → 工程化交付的可复盘项目清单。

## 4) 不确定性与盲区（这篇材料没解决但决定成败）

- **许可与专利边界**：硬件专利/图纸许可、第三方组件许可，可能引发“可复现但不可商用”的摩擦。
- **安全与责任**：高速跑步与耐久测试存在人身/财产风险；若出现事故，可能触发监管与舆论的 catastrophic 路径。
- **开源≠标准**：没有认证/一致性测试与版本治理，生态可能碎片化，复现反而变成“各自一套”。
- **价值捕获位置不清**：全栈开源会削弱传统“闭源护城河”，价值更可能转移到 QA、制造、交付、数据与认证，但材料未给出可核验路径。
